

> 一套可长期遵循的、覆盖"从想法到上线再到迭代"全生命周期的工作流。 核心原则：**每个阶段都有明确的产出物和质量门（Quality Gate），用产出物驱动流程，而不是用时间驱动。**

---

## 〇、整体认知：这不是一条直线

在开始之前，记住三件事，否则工作流会变成形式主义：

1. **它是迭代的，不是瀑布式的。** 阶段之间会回流。接口设计可能推翻数据库设计，UI 评审可能改需求。允许回退是正常的，关键是每次回退都要同步更新对应的产出物。
2. **它是可裁剪的。** 不是每个项目都要把每个步骤做满。小项目（个人/Demo）可以把多个产出物合并成一两份轻量文档；中大型项目则需要完整执行。文末有"按项目规模裁剪"的建议。
3. **产出物 > 流程本身。** 判断一个阶段是否完成，看它该有的交付物在不在、质量够不够，而不是"我花了多久"。

全流程可分为 **三大块、十个阶段**：

- **构思与设计**（阶段 1–6）：搞清楚做什么、怎么做
- **构建与验证**（阶段 7–8）：写出来、测试它
- **交付与运营**（阶段 9–10）：上线、维护、迭代

---

## 第一块：构思与设计

### 阶段 1 · 需求分析（Requirements Analysis）

**目标**：把模糊的想法变成清晰、有边界、可验证的需求。这是整个项目最关键的一步——这里出错，后面所有工作都是在错误的地基上施工。

**关键活动**：
- 需求收集：与利益相关者/用户沟通，做用户访谈或调研
- 区分**功能性需求**（系统要做什么）和**非功能性需求**（性能、安全、可用性、兼容性等——最容易被忽略，但决定项目成败）
- 编写**用户故事 / 用例**：以"作为某角色，我想做某事，以便达成某目的"的格式描述
- 需求**优先级排序**：推荐 MoSCoW 法（Must / Should / Could / Won't）
- 明确**范围边界**：写清楚"这一期不做什么"，防止范围蔓延（scope creep）
- 为每条需求定义**验收标准**（Acceptance Criteria）

**产出物**：

- 📄 需求文档 / PRD（产品需求文档）
- 📄 用户故事列表 + 优先级
- 📄 验收标准清单

**质量门（进入下一阶段的检查清单）**：

- [ ] 每条核心需求都有明确的验收标准吗？
- [ ] 非功能性需求（性能/安全/兼容性）写了吗？
- [ ] 范围边界清晰吗，知道"不做什么"吗？
- [ ] 优先级排过了吗，知道 MVP 是哪些吗？

---

### 阶段 2 · 功能与产品设计（Feature & Product Design）

**目标**：把需求转化为具体的功能结构和流程，确定"系统长什么样、怎么用"，但还不涉及视觉。

**关键活动**：

- **信息架构（IA）**：页面/模块如何组织、导航结构
- **功能模块拆分**：把大功能拆成可独立开发的小模块
- **用户流程图（User Flow）**：用户完成关键任务的路径
- **业务流程图**：后台业务逻辑的流转（如订单状态流转、审批流）
- **线框图（Wireframe / 低保真原型）**：只画布局和元素，不做美化
- **状态设计**：每个界面/功能的各种状态——空状态、加载中、错误、成功、无权限等（这是新手最常漏的，直接影响体验质量）

**产出物**：

- 📄 功能清单 / 功能模块图
- 📄 用户流程图、业务流程图
- 📄 线框图（低保真原型）

**质量门**：

- [ ] 每个功能都能追溯到某条需求吗（没有"凭空多出来"的功能）？
- [ ] 关键流程的异常分支和边界情况考虑了吗？
- [ ] 每个界面的各种状态都设计了吗？

---

### 阶段 3 · UI / UX 设计（Visual & Interaction Design）

**目标**：在线框图基础上做出高保真、可交付给开发的视觉与交互设计。

**关键活动**：

- 建立**设计系统 / 设计规范（Design Tokens）**：色彩、字体、间距、圆角、阴影等的统一标准
- 搭建**组件库**：按钮、表单、卡片、弹窗等可复用组件
- 输出**高保真设计稿**
- **交互设计**：动效、过渡、反馈、微交互
- **响应式设计**：定义断点，确定移动端/平板/桌面的适配方案
- **可访问性（a11y）**：对比度、键盘可操作、语义化、屏幕阅读器支持
- **设计评审（Design Review）**：和需求方、开发对齐

**产出物**：

- 🎨 设计系统 / 规范文档
- 🎨 高保真设计稿（Figma 等）
- 🎨 可交付的组件标注 / 切图

**质量门**：

- [ ] 设计是否复用了统一的设计系统，而不是每个页面各画各的？
- [ ] 响应式断点和各端表现定义清楚了吗？
- [ ] 各种状态（空/错/加载）都有对应视觉稿吗？
- [ ] 可访问性基本要求满足了吗？

---

### 阶段 4 · 技术选型与架构设计（Tech Stack & Architecture）

**目标**：决定用什么技术、系统怎么搭。这一步决定了项目的可维护性、性能上限和团队协作效率，**绝对不能跳过**。

**关键活动**：

- **技术栈选型**：前端框架、后端语言/框架、数据库、缓存、消息队列等。选型依据应是需求（尤其非功能性需求）和团队熟悉度，而非"哪个最新潮"
- **渲染/架构模式**：SPA / SSR / SSG / 同构？单体 / 微服务 / Serverless？
- **系统架构设计**：分层结构、模块划分、数据流向（画**架构图**）
- **第三方服务**：支付、登录、短信、对象存储、地图等的选型
- **部署架构**：服务器/云、CDN、负载均衡、环境划分
- **关键技术决策记录**：把"为什么这样选"记下来（ADR，架构决策记录）

**产出物**：

- 📄 技术选型说明
- 📄 系统架构图
- 📄 关键技术决策记录（ADR）

**质量门**：

- [ ] 技术选型能满足非功能性需求（并发、性能、安全）吗？
- [ ] 架构是否过度设计（小项目别硬上微服务）或设计不足？
- [ ] 部署和环境（开发/测试/生产）规划清楚了吗？

---

### 阶段 5 · 数据库设计（Database Design）

**目标**：设计稳定、规范、可扩展的数据结构。数据库一旦上线积累了数据，改起来代价极高，所以要尽量一次设计到位。

**关键活动**：

- **概念模型**：画 **ER 图**（实体—关系图），理清实体和关系
- **逻辑模型**：应用**范式**（通常到第三范式 3NF），合理时再做反范式优化
- **物理模型**：确定表、字段、类型、主外键、约束
- **索引设计**：根据查询场景设计索引，平衡读写性能
- **数据字典**：每张表、每个字段的含义和约定
- **迁移策略**：用 migration 工具管理表结构变更（不要手动改库）

**产出物**：

- 📄 ER 图
- 📄 数据库表结构 / 建表脚本
- 📄 数据字典
- 📄 数据库迁移脚本

**质量门**：

- [ ] 表结构能支撑所有功能的数据需求吗？
- [ ] 关键查询有对应索引吗？是否有明显性能隐患？
- [ ] 字段命名、类型规范统一吗？
- [ ] 用 migration 管理变更了吗？

---

### 阶段 6 · 前后端接口设计（API Design）

**目标**：定义前后端之间的"契约"。接口先定好，前后端就能并行开发，这是提效的关键。

**关键活动**：

- **接口风格**：REST / GraphQL / RPC
- **接口规范**：统一的 URL 命名、请求方法、请求/响应数据结构
- **统一返回格式**：约定 `code / message / data` 等结构
- **错误码设计**：系统化的错误码体系
- **认证与授权**：JWT / OAuth / Session，以及权限模型
- **版本控制**：API 版本策略（如 `/v1/`）
- **接口文档**：用 **OpenAPI / Swagger** 等规范化描述
- **Mock 数据**：提供 mock 接口，让前端不必等后端

**产出物**：

- 📄 接口文档（OpenAPI/Swagger）
- 📄 统一返回格式与错误码规范
- 📄 Mock 数据 / Mock 服务

**质量门**：

- [ ] 接口文档完整、前后端都认可这份"契约"吗？
- [ ] 返回格式、错误码、鉴权方式统一规范了吗？
- [ ] 前端能基于 Mock 独立开发吗？

---

## 第二块：构建与验证

### 阶段 7 · 开发与协作（Development）

**目标**：高质量地把设计变成代码，并保证团队协作有序。

**关键活动**：

- **项目脚手架搭建**：统一工程结构、依赖、配置
- **代码规范**：Lint（ESLint 等）、格式化（Prettier 等）、命名约定，最好用工具强制
- **Git 工作流**：分支策略（如 Git Flow / Trunk-Based）、提交信息规范（如 Conventional Commits）
- **环境配置**：开发 / 测试(staging) / 生产环境分离，配置用环境变量管理
- **前后端并行开发**：基于阶段 6 的接口契约和 Mock
- **代码审查（Code Review / PR）**：合并前必须有人审查
- **持续提交与集成**：小步提交，频繁集成

**产出物**：

- 💻 符合规范的源代码
- 📄 README / 开发文档（如何启动、构建、贡献）
- 📄 代码规范配置文件

**质量门**：

- [ ] 代码通过 Lint 和格式检查了吗？
- [ ] 每个 PR 都经过 Review 了吗？
- [ ] 敏感配置（密钥等）有没有被硬编码进代码/提交进仓库？

---

### 阶段 8 · 测试（Testing）⚠️ 决定质量的关键，常被个人开发者忽略

**目标**：在上线前发现问题。没有测试的"完成"只是"自以为完成"。

**关键活动**：

- **单元测试**：测试函数/组件的最小逻辑单元
- **集成测试**：测试模块之间、接口与数据库的协作
- **端到端测试（E2E）**：模拟真实用户操作完整流程
- **测试用例**：直接从阶段 1 的验收标准转化而来
- **专项测试**：性能测试、安全测试、兼容性测试（浏览器/设备）
- **回归测试**：改动后确保旧功能没被破坏

**产出物**：

- 📄 测试用例
- 💻 自动化测试代码
- 📄 测试报告 / 缺陷列表

**质量门**：

- [ ] 核心功能都有测试覆盖吗？
- [ ] 所有验收标准都被测试验证了吗？
- [ ] 已知严重 bug 都修复了吗？

---

## 第三块：交付与运营

### 阶段 9 · 部署与上线（Deployment / DevOps）

**目标**：把代码安全、可重复地发布到生产环境。

**关键活动**：

- **CI/CD 流水线**：自动化构建、测试、部署
- **容器化**：如 Docker，保证环境一致
- **环境管理**：测试环境先行验证，再发布生产
- **发布策略**：灰度发布 / 蓝绿部署 / 滚动更新，降低风险
- **回滚预案**：出问题能快速回退
- **配置监控与日志**：上线即接入监控、报警、日志收集

**产出物**：

- 📄 部署文档 / 流程
- 💻 CI/CD 配置、Dockerfile 等
- 📄 上线检查清单（发布前/后）

**质量门**：

- [ ] 能一键、可重复地部署吗（而不是手动操作一堆步骤）？
- [ ] 出问题能快速回滚吗？
- [ ] 监控、日志、报警都接好了吗？

---

### 阶段 10 · 运维与迭代（Maintenance & Iteration）

**目标**：项目上线不是终点，而是迭代循环的起点。

**关键活动**：

- **监控与告警**：盯住性能、错误率、可用性
- **日志分析**：定位问题、洞察用户行为
- **收集用户反馈**：作为下一轮需求输入
- **性能优化**：基于真实数据持续优化
- **Bug 修复与安全更新**：及时响应
- **迭代规划**：把反馈和新需求带回**阶段 1**，开启下一轮循环

**产出物**：

- 📄 监控/告警配置
- 📄 迭代需求池 / 路线图
- 📄 复盘文档

**质量门**：

- [ ] 系统健康状况可观测吗（不是用户报障了才知道出事）？
- [ ] 用户反馈有渠道收集并进入下一轮规划吗？

## 附：如何按项目规模裁剪这套流程

这套流程是"完整版"。实际使用时按规模裁剪，避免小项目被流程拖累：

|阶段|个人 / Demo 项目|中型项目|大型 / 团队项目|
|---|---|---|---|
|1 需求分析|一份简单需求清单|完整 PRD + 用户故事|完整 PRD + 调研|
|2 功能设计|简单线框/手绘|流程图 + 线框图|完整产品设计|
|3 UI 设计|可用现成 UI 库|高保真稿|完整设计系统|
|4 技术架构|**必做**（哪怕只是想清楚）|**必做**|详细架构 + ADR|
|5 数据库设计|**必做**|**必做**|**必做**|
|6 接口设计|简单约定即可|**完整接口文档**|完整文档 + Mock|
|7 开发规范|Lint + Git 基础|**完整规范 + Review**|严格规范|
|8 测试|核心功能手测|**关键路径自动化**|完整测试体系|
|9 部署|简单脚本即可|**CI/CD**|完整 DevOps|
|10 运维迭代|基础监控|**监控 + 反馈**|完整可观测性|

**永远不能省的核心（无论项目多小）**：需求边界（阶段1）、技术选型（阶段4）、数据库设计（阶段5）、基础的版本控制与测试（阶段7-8）。这四块是质量和返工成本的根源。

## 核心心法总结

1. **用产出物驱动流程**：每个阶段做完都该有可交付的东西，否则就是没做完。
2. **接口先行**：阶段 6 定好契约，前后端并行，这是单点提效最大的地方。
3. **质量门是闸门**：不达标不放行，把问题挡在早期（越晚发现问题，修复成本越高，呈指数级增长）。
4. **流程是迭代环，不是直线**：阶段 10 的反馈回到阶段 1，循环往复。
5. **可裁剪不可跳过核心**：小项目轻量化执行，但需求、架构、数据库、测试这几块的"思考"不能省。
