### OSI七层网络模型，简要介绍一下

- **物理层：**定义物理设备标准，传输比特流。
- 数据链路层：封装成帧、透明传输、差错检测
- 网络层：数据包的封装传输。路由选择，流量控制，错误处理和诊断
- 传输层：可靠的**端到端**（进程）传输、流量拥塞控制、差错控制（TCP/UDP）
- 会话层：传输层的基础上建立连接和管理对话，包括对话管理，令牌控制，同步控制
- 表示层：管理数据结构的传输、数据的解释、加密、解密、压缩等处理
- 应用层：具体应用如FTP文件传输、Telnet服务、HTTP服务、DNS，SNMP邮件服务等

### TCP/IP四层模型

TCP/IP四层模型是为网络通信定义的一组协议层次，侧重于数据从源端到目的端的传输。

- **网路接口层：**负责物理传输介质的管理，相当于物理层和数据链路层。包括Ethernet（以太网协议）、FDDI（光纤分布数据接口）、ATM（异步传输模式，适用于高速数据传输）、PPP（点对点协议）
- **网络层：**数据包的路由，传输和目标地址解析。包括IP协议（负责选择路由、分段和重组数据包）、ARP（地址解析协议将IP地址转化为MAC地址））、ICMP（诊断网络故障）
- **传输层：**端到端通信的可靠性保证、流量控制、多路复用、优先权和安全性控制。主要包括TCP 和UDP
- **应用层：**定义实际应用程序之间的通信规则。Telnet（远程登录）、FTP（文件传输）、SMTP（电子邮件传输）、DNS（域名解析）、HTTP、NNTP(新闻组消息传输)。保证可靠性、流量控制和差错检测

### HTTP原理

HTTP是一个无状态协议。无状态指client和server之间不需要建立持久的连接，发出请求-返回响应后连接就关闭了。服务器端不保留连接的有关信息

### 从浏览器地址输入url到显示主页的过程

1. 浏览器通过域名查找对应的ip地址。通过DNS解析服务器域名得到主机ip地址
    
2. 浏览器与服务器通过三次握手建议TCP连接
    
3. 浏览器向服务器发送一个HTTP请求
    
4. 服务器处理请求（处理请求参数、cookie，生成HTML响应）
    
5. 服务器返回一个HTML响应
    
6. 浏览器解析渲染页面
    
7. TCP四次挥手，结束
    

### HTTP/1.0, 1.1, 2.0区别

HTTP/1.0：默认是短连接，每次请求都要重新建立TCP连接 开销大

HTTP/1.1：默认为长连接 一个TCP连接可以复用，减少握手开销，响应按顺序返回

HTTP/2.0：多路复用，请求响应无需排序。做了Header压缩，服务器推送等优化

### TCP的粘包和拆包

TCP是面向流，没有界限的一串数据。TCP底层并不了解上层业务数据的具体含义，它会根据TCP缓冲区的实际情况进行包的划分，所以在业务上认为，一**完整的包可能会被TCP拆分成多个包进行发送，也有可能把多个小的包封装成一个大的数据包发送**，这就是所谓的TCP粘包和拆包问题

粘包：发送的数据小于发送缓冲区大小

拆包：发送的数据大于发送缓冲区/最大报文长度

会导致协议解析消息困难，应用层需要处理消息界限

解决方案：定长消息、分隔符、长度前缀（标识消息长度）——应用层处理

### ARP协议工作过程

Address resolution protocol，地址解析协议。用于IP地址到MAC地址的映射。

1、源主机发送数据包时，检查自己的ARP列表是否存在IP地址对于的MAC地址，如果有则直接发

2、如果没有则发起ARP请求的广播包，查询目的主机对应的MAC地址。

3、网络中其他主机收到请求后检查IP地址，如果不相同则忽略，如果相同则返回自己的IP地址和MAC地址

4、源主机收到响应数据包后将MAC地址添加到自己的ARP列表中

### 什么是HTTPS

HTTP被用于再Web浏览器和网站服务器之间传递信息，以明文方式发送内容，如果攻击者截取传输报文可以直接读取信息。

HTTPS加入了SSL协议，依靠证书来验证服务器身份，并对浏览器和服务器之间的同i性能进行数据加密

**加密流程：**

1. 发起请求： 客户端在通过TCP 和服务器建立连接之后（ 443 端口），发出一个请求证书的消息给服务器，在该请求消息里包含自己可实现的算法列表和其他需要的消息。
    
2. 证书返回：服务器端在收到消息后回应客户端并返回证书，在证书中包含服务器信息、域名、申请证书的公司、公钥、数据加密算法等。
    
3. 证书验证： 客户端在收到证书后，判断证书签发机构是否正确，并使用该签发机构的公钥确认签名是否有效， 客户端还会确保在证书中列出的域名就是它正在连接的域名。如果客户端确认证书有效，则生成对称密钥，并使用公钥将对称密钥加密。
    
4. 密钥交换： 客户端将加密后的对称密钥发送给服务器，服务器在接收到对称密钥后使用私钥解密
    
5. 数据传输： 经过上述步骤，客户端和服务器就完成了密钥对的交换， 在之后的数据传输过程中， 客户端和服务端就可以基于对称加密（ 加密和解密使用相同密钥的加密算法）对数据加密后在网络上传输，保证了网络数据传输的安全性。

## HTTP/HTTPS通信流程

### **HTTP**

**1、建立连接 (TCP 三次握手)**：

- **客户端 -> 服务器**：“你好，我想和你建立连接。” (SYN)
- **服务器 -> 客户端**：“好的，我准备好了。这是我的确认。” (SYN-ACK)
- **客户端 -> 服务器**：“收到，我们开始吧。” (ACK)

**2、发送请求 (Request)**：

- 客户端沿着这条电话线，发送一个**纯文本**的 HTTP 请求报文。

**3、接收响应 (Response) 并关闭**：

- 服务器收到请求，处理完毕后，发回一个**纯文本**的 HTTP 响应报文。
- 浏览器收到 HTML，解析并展示。
- 通信结束，TCP 连接关闭（HTTP/1.1 可能会保持连接 Keep-Alive，但本质不变）。

### **HTTPS**

1、**建立连接 (TCP 三次握手)**：

2、**建立加密通道 (SSL/TLS 握手)**：  
**目标**：客户端和服务器必须在**不被窃听**的情况下，商量出一个**“对称加密密钥”（我们叫它“会话密钥 Session Key”）**，这把钥匙将被用来加密后续所有的 HTTP 数据。

**3、发送加密请求 (Encrypted Request)**：

- 客户端把 HTTP 请求报文准备好。
- 然后，用第 2 步商量好的那个**“会话密钥”，对这个报文进行**对称加密**。
- 加密后的结果（一堆乱码）发送给服务器。

4、**接收加密响应 (Encrypted Response) 并关闭**：

- 服务器收到“保险箱”，用自己手里**相同**的那个**“会话密钥”**来解密，得到明文的 HTTP 请求。
- 服务器处理完，把 HTTP 响应报文（`<html>...`）也用这个**“会话密钥”**加密，发回给客户端。

### **SSL/TLS**

一个安全层，专门负责在“TCP 三次握手”之后、在“HTTP 请求”之前，建立一个**加密通道**。

握手过程：

- 客户端 → 服务器 发送 随机数A + 加密算法列表
- 服务端 → 客户端 发送 选定的加密算法 + 随机数B + 数字证书（公钥）
- 客户端 → 服务端 验证证书合法 发送 随机数C（公钥加密）
- 服务器使用私钥解密
- 使用随机数A\B\C 通过一个相同的算法混合计算出 会话密钥

随机数A+B的作用是为了保证每一次握手的时效性和唯一性

随机数C保证机密性
